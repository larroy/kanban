# Architecture

## Stack

| Layer | Choice |
|---|---|
| Framework | SvelteKit 2 + Svelte 5 (runes mode) |
| Language | TypeScript |
| Build | Vite 7 (via SvelteKit) |
| Styling | TailwindCSS v4 (`@tailwindcss/vite`) |
| ORM | Drizzle ORM + `pg` (node-postgres) |
| Drag & Drop | SortableJS (via a Svelte action wrapper) |
| Adapter | `@sveltejs/adapter-node` |

SvelteKit serves both the frontend and the API — no separate backend process.

## Database Schema

```sql
users       (id, name, color, created_at)
projects    (id, name, description, status, created_at, updated_at)
tasks       (id, title, description, status, position, project_id → projects, assignee_id → users, created_at, updated_at)

INDEX tasks_status_position_idx ON tasks (status, position)
INDEX tasks_project_id_idx      ON tasks (project_id)
```

`status` columns use check constraints: projects accept `active | done`; tasks accept `todo | doing | done`.

### Triggers (in `drizzle/triggers.sql`)

- **`set_updated_at`** — `BEFORE UPDATE` on both `tasks` and `projects`; sets `updated_at = now()`.
- **`check_project_completion`** — `AFTER INSERT | UPDATE | DELETE` on `tasks`; automatically sets the parent project's `status` to `done` when every task in the project is `done`, and back to `active` otherwise.

## Data Flow

### Server-side rendering

1. `+layout.ts` fetches `/api/users` — result available as `data.users` on every page.
2. `+page.server.ts` (board) runs `getTasks()` and `getProjects()` directly via Drizzle, returning typed data to the page.
3. SvelteKit serialises the data and hydrates the client.

### Client-side board state (`src/lib/board.svelte.ts`)

Module-level Svelte 5 `$state` — one shared reactive store for all board components:

```
board.tasks        raw list (all statuses)
board.todo         $derived: tasks where status='todo', sorted by position
board.doing        $derived: tasks where status='doing', sorted by position
board.done         $derived: tasks where status='done', sorted by position
board.moveTask()   optimistic update → PATCH /api/tasks/:id/move → rollback on error
```

`KanbanBoard` calls `board.setTasks(initialTasks)` once on mount. Subsequent mutations (add, edit, delete, move) update the store directly without a full page reload.

### Drag and drop

`src/lib/sortable.ts` is a ~15-line Svelte action that wraps SortableJS:

```
use:sortable={{ group: 'kanban', onEnd }}
```

`KanbanColumn` attaches this action to its task list container. On drop, `onEnd` reads the task id from `data-id` and the destination column from `data-status`, then calls `board.moveTask()`.

### Move endpoint (`PATCH /api/tasks/:id/move`)

Runs inside a single PostgreSQL transaction using a raw `pg` client (bypassing Drizzle for atomicity):

1. `SELECT ... FOR UPDATE` — locks the row.
2. **Same column**: shifts positions of cards between old and new index.
3. **Different column**: compacts the gap in the source column, opens a slot in the destination.
4. Updates the task's `status` and `position`.
5. `COMMIT` — or `ROLLBACK` on any error.

## API Reference

```
GET  /api/users                 → User[]
POST /api/users                 → User          { name, color? }

GET  /api/projects              → ProjectWithStats[]
POST /api/projects              → Project       { name, description? }
GET  /api/projects/:id          → ProjectWithStats & { tasks }
PATCH /api/projects/:id         → Project       { name?, description? }
DELETE /api/projects/:id        → 204

GET  /api/tasks                 → TaskWithAssignee[]   ?projectId  ?status
POST /api/tasks                 → Task          { title, projectId, description?, status?, assigneeId? }
GET  /api/tasks/:id             → TaskWithAssignee
PATCH /api/tasks/:id            → Task          { title?, description?, assigneeId? }
DELETE /api/tasks/:id           → 204

PATCH /api/tasks/:id/move       → Task          { status, position }
```

## Configuration

**`svelte.config.js`**
- Adapter: `@sveltejs/adapter-node`
- Alias: `$components → src/components`
- Runes mode forced for all non-node_modules files

**`vite.config.ts`**
- Plugins: `tailwindcss()` (must come before `sveltekit()`)

**`drizzle.config.ts`**
- Schema: `drizzle/schema.ts`
- Output: `drizzle/migrations/`
- Connection: `DATABASE_URL` env var (falls back to Unix socket)

**`.env`**
```
DATABASE_URL=postgres:///kanban?host=/var/run/postgresql
```
Peer authentication via Unix socket — no password required for the local `piotr` user.

## Development

```bash
npm run dev        # dev server on localhost:5173
npm run build      # production build → build/
npm run preview    # preview production build

# DB operations
npx drizzle-kit generate          # generate migration from schema changes
npx drizzle-kit migrate           # apply pending migrations
psql -d kanban -f drizzle/triggers.sql   # (re-)apply triggers
```
